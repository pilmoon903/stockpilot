# ADR-005: Forward Testing 채택 (백테스팅 배제)

**날짜:** 2026-04-16
**상태:** Accepted
**관련:** PRD v1.5 섹션 10, 15, 16 (KPI·리스크·확장 로드맵)

---

## 컨텍스트

전통적 퀀트 트레이딩 시스템은 실전 투입 전 **백테스팅**(과거 데이터로 전략 시뮬레이션)으로 성능을 검증한다. 개인 퀀트 투자자도 이 패턴이 표준이다.

StockPilot도 실전 자금 투입 전 검증 단계가 필요하다. 문제는 우리 시스템이 **LLM + HITL 기반**이라 전통 백테스팅이 구조적으로 성립하지 않는다는 점이다.

### 백테스팅이 우리 시스템에 맞지 않는 이유

1. **LLM Look-ahead bias 구조적 발생**
   Claude는 2026년 시점 지식을 학습. 2020년 공시를 주고 "그때 판단을 시뮬레이션"해도 이미 결과를 아는 상태에서 분석. 데이터 정제·프롬프트 엔지니어링으로도 완전 제거 불가.

2. **HITL 재현 불가**
   "과거 시점의 내가 승인 눌렀을까?"는 본질적으로 시뮬레이션 불가. 당시 심리·감정·뉴스 환경 복원 X.

3. **소표본**
   관심 종목 12개, 하루 0~3건 매매. 1년 관측해도 유의미한 통계 표본 확보 어려움. 백테스팅 결과의 통계적 유의성 보장 불가.

4. **AI 분석 방식이 매 버전 변경됨**
   프롬프트·모델·컨텍스트 주입 방식이 계속 진화. 과거 시점으로 "고정된 전략"이 존재하지 않음. 백테스팅 대상 자체가 움직이는 표적.

---

## 검토한 대안

| 대안 | 장점 | 단점 |
|---|---|---|
| **Forward Testing (정방향 실관측)** | Look-ahead bias 없음, 실전 환경 검증, HITL 자연 재현 | 시간 소요 (10영업일+), 표본 제한 |
| 전통 백테스팅 (AI 포함) | 업계 표준, 빠른 검증 | Look-ahead bias 100%, 의미 없는 결과 |
| 룰 부분만 백테스팅 (AI 제외) | 스크리닝 룰 건전성 확인 | AI 분석 부분은 여전히 미검증 |
| 외부 백테스팅 도구 연동 (Backtrader·QuantConnect) | 성숙한 프레임워크 | AI 분석 시뮬레이션 불가, 연동 비용 > 가치 |
| 검증 생략 | 시간 절약 | 실전 투입 리스크 과대 |

---

## 결정

**Forward Testing을 MVP의 주 검증 방식으로 채택. 백테스팅은 Phase 2에 "룰 스크리닝 부분만" 제한적으로 도입 검토.**

### 근거

1. **Look-ahead bias 원천 차단**
   Forward Testing은 "오늘 이후" 시점 데이터로만 검증. LLM이 미래 결과를 알 수 없는 구조. 가장 정직한 검증 방법.

2. **실전 조건 그대로 검증**
   모의투자 API → 실전 API 전환 시 파이프라인 변경 없음. 모의 단계에서 검증된 것이 곧 실전에서 작동. 백테스팅과 실전 간 괴리 없음.

3. **HITL 자연 재현**
   "오늘 AI가 준 브리핑을 보고 내가 내리는 결정"이 Forward Testing의 본질. HITL 구조가 시뮬레이션 없이 그대로 동작.

4. **이벤트 수집·KPI 측정 일관성**
   PRD 9조 이벤트 7종과 10조 AARRR·Decision Quality Lift는 실시간 운영에서 자동 수집됨. 백테스팅과 달리 "지표 산출 방식"이 MVP와 동일.

5. **MVP 철학 부합**
   Lean 원칙상 "만들기 위한 만들기 금지". 백테스팅 엔진 구현은 10~15h + 결과 신뢰성 낮음. Forward Testing은 이미 PRD에 포함된 Day 8~9 + 모의 4주 관측으로 자연 해결.

---

## 검증 단계 (3단계 전환 프로토콜)

| 단계 | 기간 | 조건 |
|---|---|---|
| Stage 1: 모의 테스트 | Day 8~9 (3~5영업일) | A/B 실험 1개 + 이벤트 수집 시작 |
| Stage 2: 모의 관측 확장 | MVP 완료 후 4주 | Decision Quality Lift 안정화 확인 |
| Stage 3: 실전 소액 | Stage 2 성공 시 | 10만원/일 상한으로 시작 |

### Stage 2 → 3 전환 기준
- Decision Quality Lift > 0 (baseline 대비 +)
- Guardrail Breach Rate 0
- False Positive Cost 평균 regret_score < 2.5
- Chaos 체크리스트 2종 발동 이력 없음

---

## 결과·영향

**긍정**
- 가장 정직한 검증 방식 (Look-ahead bias 0)
- 백테스팅 엔진 구현 비용 절감 (10~15h)
- 실전 전환 시 파이프라인 변경 없음
- KPI 측정 프레임 일관성

**부정**
- 검증에 시간 소요 (최소 4주)
- 소표본 한계는 여전히 존재 (관측 누적으로만 해소)
- "백테스팅 안 했다" 질문에 추가 설명 필요

**후속 작업**
- Day 8~9에 A/B 실험 1개 + 이벤트 수집 시작
- MVP 완료 후 주간 복기로 Stage 2 관측 추적
- Phase 2에 "룰 스크리닝만 백테스팅" 검토 (AI 제외)

---

## 재평가 조건

- Phase 2에 룰 스크리닝 백테스팅 결과가 Forward Testing과 상충 → 방법론 재점검
- Forward Testing 관측 기간이 6개월 넘었는데도 통계적 유의성 부족 → 표본 확대 방법 검토
- LLM이 "과거 시점 지식 제한" 기능 제공 시 → 백테스팅 재검토

---

## 참고: 백테스팅 vs Forward Testing 차이

| 항목 | 백테스팅 | Forward Testing |
|---|---|---|
| 데이터 방향 | 과거 → 현재 | 현재 → 미래 |
| Look-ahead bias | 발생 (LLM은 특히 심함) | 원천 차단 |
| 실행 시간 | 몇 분 ~ 몇 시간 | 몇 주 ~ 몇 개월 |
| 시장 조건 재현 | 완벽하지 않음 | 실제 그 자체 |
| HITL 재현 | 불가능 | 자연 발생 |
| AI 시스템 적합성 | 낮음 | 높음 |

**핵심:** 전통 퀀트엔 백테스팅이 최적, **LLM·HITL 시스템엔 Forward Testing이 최적**. 시스템 특성에 맞는 검증 방식 선택이 PM 판단.

---

## 면접 답변 프레임 (참고)

**Q. 왜 백테스팅 안 했나요?**

A: "전통 퀀트는 백테스팅이 표준이지만 LLM 기반 시스템엔 구조적으로 맞지 않습니다. Claude는 현재 시점 지식으로 과거 공시를 보기 때문에 Look-ahead bias가 100% 발생하고, HITL 구조상 과거 시점의 인간 결정을 재현할 수도 없습니다. 대신 Forward Testing — 모의투자 API로 실시간 운영하며 검증하는 방식 — 을 채택했고, 10영업일 + 모의 4주의 3단계 전환 프로토콜로 실전 소액 투입 기준을 사전 정의했습니다. **시스템 특성에 맞는 검증 방식을 선택하는 것이 PM의 판단이라고 생각합니다.**"
